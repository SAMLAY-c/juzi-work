## 2.开始执行：

**好的启动，不会把问题留在未来**

### 1. Kickoff Meeting：

1. **向****所有人****同步项目背景、目标、及相关信息**

2. 明确分工、时间要求及产出要求，并获得**【确认】**

3. **确认项目的沟通机制：**

4. 项目群
5. 日会/周会/临时会议/
6. 汇总信息的方式（几个文档？分别是什么目标定位，编辑权限分配）

7. **项目风险点预估：**有哪些不可预见的、可能存在的问题？

8. 技术上可能不可行

9. 因为没有相关经验，时间上可能超期

### 2. 领走了任务，下一步

**按时的推动及有质量的交付是唯一衡量标准**

#### 1. 每一个里程碑时间点前，【提前】确认是否可按期完成

1. 所有的延期，都来自于没有过程中的确认

2.一个长时间的里程碑，需要增加多个确认节点

3.如果有必要，确认节点的产出也需要明确

#### 2. 有了变动，怎么办？

1. 需求的变动、时间要求的变动、成员的变动、项目延期的变动

2. 向所有成员同步所有信息！**必须！必须！必须！**

3. 按计划进行中，也需要同步及确认

4.有变动，需要决策的，快速拉会：站会、电话会、扭头沟通一嘴…

5.核心是：拿到**结论+计划+信息同步所有人并确认**

#### 3. 卡住了，怎么办？

1. 技术不可实现、需要对方配合却无响应、需要决策却没人拍板…

2. **动用一切你可以动用的资源**：你的leader、对方的leader、双方的老板…

3. 继续与不继续？改目标or保持目标？ **结论+计划+信息同步所有人并确认**

#### 4. 沟通的基本框架：

1. **问题信息：**订单数据只有部分能匹配

2. **问题原因：**unionID与群内数据对不上

3. **预期解法：**方法1、方法2、方法3，各自的优劣势

4. **需要谁来做什么：**决策接受哪个方法

#### 5. 会议纪要的重要性！

1. **记录每一次的沟通结论：**变动信息、决策思路及结论、后续任务

2. 结论，需要相关人**【确认】**

3. 后续任务分配到人头，**并【确认】**

4. **留档备查、复盘使用**

## 一些原则：

### 1. 项目负责人：

1. **必须、一定、只能是【唯一一个】**

2. ta必须是掌握**所有项目信息**的人

3. ta必须是**参与所有关键角色沟通**的人

4. ta必须有**调度项目成员支持**的权限

5. 项目成员不配合，ta必须**有权限和相关的管理者**进行沟通

### 2. 必须杜绝多头沟通

1. 项目负责人作为唯一沟通接口人

2. 需要多方沟通确认的问题，由项目负责人发起

3. 多头沟通，是信息不同步的罪魁祸首

### 3. 永远保持信息同步：

1. 让每个人知道我们现在在什么状态，下一步要做什么

2. 同步信息的核心，在于让相关方**【收到】**信息。

3. **提一嘴或发一条信息，不叫同步**

4. 绝对不能指望一条@解决问题

5. 不要在沟通过程中同步信息，同步最终决策逻辑和结论！

### 4. 不能坐着等结果，而要管理过程

1. 项目延期，总是在里程碑当天发现

2. 保持沟通的频率，**轻度、多次、点对点**的沟通，**团队内同步**

3. **对不可信的项目节点/人员，要做更多的确认动作**

4. 时间上的确认

5. 已产出内容的确认
