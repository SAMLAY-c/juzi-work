## 特殊功能需求

1. **质检机制（全链路）** 对最终输出不利于兴趣岛品牌/课程的话术、平台负面言论、与历史重复、乱码异常（含 Base64/Hex/Hash 等编码串）、英文占比异常、思考过程/系统提示语泄露、历史相似度等检测。、
    
2. **质检不通过的二次生成** 当质检失败，基于事件机制将该消息**重新进入流程引擎**，触发新一轮生成，再次质检，最多两次，失败则转人工。
    
3. **静默时段与 06:00 延迟再生成** 接量期**22:00 后进入静默**：用户消息以事件入队，**次日 06:00** 自动再生成。为避免时段错配（如清晨输出“晚上好”），需携带**时间语义标签**参与话术生成。
    
4. **当日与次日事件**
    
      由于流程引擎定时事件的特殊性，需要区分当日与次日逻辑，如静默时间为当日22:00到次日早上六点，则用户输入在22:00-0:00时，事件设置为次日6:00触发，而在0:00-6:00输入，事件设置为当日6:00触发

## 沟通技巧

1. 关键人物与情绪管理
    

- 针对AI各项目推进负责人（非最高决策）
    
    - 沟通策略：提供足够“被重视感”和过程透明；不主动披露任何“并行公司其他项目”的进展与细节。
        
    - 建议参考话术：
        
        - “兴趣岛是我们的重点客户在我们内部排期中优先级靠前——”
            
        - “问题原因和解决方案在调整后会第一时间同步给您——”
            

2. 人员与口径一致性
    

- **PE 同学人员储备**：保持统一口径，避免个别表述造成“人手不稳”的印象。
    

3. **Badcase 沟通原则**
    
    1. 业务方**不关心技术细节**，更在意**如何杜绝再次发生**。
        
    2. 对外口径：**少描述过程，用简答易懂的方式让业务明白原理，给出防止复发方案，**不要回避问题。
        
    3. **调性控制**：保持信心与克制，不夸大（例如：避免“效果非常好”），使用中性指标化表达（“已达成预期/按原计划推进”）。
        
4. **Badcase 应对办法**
    
    1. **确认知悉**：第一时间回复“已收到—已在处理—给到预计处理时间”。
        
    2. **原因结论**：换位思考用业务能听懂的语言给出问题的**原因**。
        
    3. **修复动作**：说明**修复方案 + 环境（测试/正式）**。
        
    4. **防止复发措施**：新增规则/质检项/开关策略/SOP 更新点，以列表呈现并**可验证**。
        
5. **问题反馈格式**
    

- 业务方常以**截图**方式反馈且难以完全转到表格；在强调后仍未按表格提交时，**退而求其次**，要求最小必填格式如下（可复制粘贴）：
    
    - **截图**：清晰展示异常位置/上下文
        
    - **问题描述**：期望和实际
        
    - **类型**：Badcase / 新需求（二选一）
        
    - **出现时间**：DD HH:mm
        
    - **复现场景**：（聊天历史）
        
    - **环境**：测试服 / 正式服
        

6. **提前沟通排期**
    

- 在实际搭建过程中，可能出现各类不可预见情况（内部或外部）。无论是测试周期拉长、流程搭建工作量远超预期，还是因并行任务过多导致资源不足而产生延期，都应第一时间评估影响，并提前与客户同步排期调整方案（原因、影响范围、应对措施与最新交付时间），避免信息滞后造成误解与信任损耗。
    

7. **文档留痕**
    

- 沟通过程必须形成可追溯记录，覆盖关键环节与决策依据，包括但不限于：新需求新增与确认、bad case 调优过程与结果、周会复盘结论与行动项、责任人及截止时间等。通过留痕确保信息一致、责任清晰、复盘可用，为后续迭代与交付风险控制提供依据。

## 客户背景

兴趣岛是一家专注于兴趣技能在线教育的公司，当前重点推进太极课程业务线，采用“先导课（低价/免费）→ 正价课转化”的商业模式，分两个核心阶段：

- 接量期：用户初次接触课程，重点在于引导客户上体验课，低体验门槛（如免费试听、9.9元入门课）
    
- 行课期：用户进入学习周期，需提升完课率、收集反馈，对用户练习的情况进行建议分析，推动购买正式课。

##### 重点问题解析及解决方案

1. **不转人工不应答**
    

**问题概述：**

- **Cloudflare服务异常**：11月18日，Cloudflare服务出现故障，导致网络瘫痪，集中爆发500系列报错。
    
- **Gemini模型厂商异常**：Gemini模型在调用过程中出现500系列报错。
    

**解决方案：**

- 为防止重复问题，技术层面增加500系列**报错重试机制**。目前日志中未再出现相关问题。
    

---

2. **模型识别有误**
    

**问题概述：**

- 在用户输入异常或语音条不清晰时，模型对特殊情况及语境的理解效果不佳，导致误识别。
    

**解决方案：**

- 调整**上下文重写模型**，增强模型对模糊意图的推断能力，目标尽量减少此类问题的出现，提高识别准确性。
    

---

3. **接量期异常破冰**

##### 重要问题原因分析

1. 模型异常 上下文重写节点出现模型输出异常，导致用户的最新输入未能正确传入后续流程引擎，进而影响整体回复效果，表现为回复不完整或不符合上下文。
    
2. 合并消息异常 针对重复应答、破冰话术重复等问题，根本原因在于消息合并的配置。该逻辑的原始设计是：为避免模型在短时间内多次重复处理消息，系统会在 20 秒内对新进线消息进行合并，并在合并后重新计时。
    
    1. 实际问题出现在以下情况：当用户两次发送消息的间隔超过 20 秒时，系统会生成两个（或更多）独立的合并消息。这同时会触发质检模型对多段内容分别进行生成，导致以下现象：
        
        1. 二次生成同类内容
            
        2. 多次重复破冰
            
        3. 重复回答同一问题
            
    2. 解决方案：将合并消息从 20 秒调整为 80 秒。80 秒的时长可以保证单轮消息流程已经完整跑完并发送成功，再进入下一轮，因此在常规交互节奏下，此类由合并策略引发的重复回复问题不会再出现。

### badcase反馈

#### 重点问题分析

##### **问题一：消息高峰期进线异常**

**1. 表现** 连续两晚（10月13日与10月14日晚）出现大模型报错、不应答及消息乱发等异常现象。

**2. 时间线**

- 10月13日晚服务期第一天课后：首次出现消息量激增导致的不稳定情况。
    
- 10月14日晚第二天课后：问题再次出现，异常更为明显。
    

**3. 影响范围** 服务期与销转期正在使用AI自动应答的多个业务群与私聊场景均受影响，部分学员出现消息延迟、丢失或内容异常。

**原因**

- **实际原因**：群聊与私聊消息在高峰期同时被处理，造成服务器并发压力过大。
    
- **本质原因**：部分业务人员将机器人回复模式设置为“自定义模式”，导致AI在群聊中无差别响应，显著增加系统负载。
    

**解决方案**

- 通知并指导所有使用AI账号的业务人员，将回复模式统一切换为“自定义模式”，群聊场景中仅保留“@托管账号”触发回复；
    
- 后续通过配置校验机制防止类似误设再次发生。
    
- 将服务器扩容以更好的应对高峰期消息量激增的情况。
    

---

##### **问题二：次日激活话术发送时间不一致**

**1. 表现** 部分学员的“次日激活话术”未在规定的触达时间统一发送，而出现于36分、41分甚至52分等不同时间点。

**2. 时间线** 主要集中于10月9日前加好友的用户（链路调整前），后续延伸影响至10月13日–14日周期。

**3. 影响范围** 少量服务期学员的激活提醒错过最佳触达时段，影响触达一致性与体验连贯性。

**原因**

- **实际原因**：早期链路中，这类用户未纳入13:00刷新机制。
    
- **本质原因**：刷新逻辑与好友建档时间存在依赖关系，旧链路未完全覆盖历史数据。
    

**解决方案**

- 调整刷新机制，确保所有用户（包括旧链路接入者）统一在13:00触发；
    
- 与高并发处理优化同步执行，防止消息挤压造成的延迟或重复发送。
    

---

##### **问题三：10月15日上午服务波动与不回复**

**1. 表现** 10月15日上午出现大规模不回复及延迟情况，部分接口报错（显示429限流）。

**2. 时间线** 10月15日 11:30—12:00为主要受影响时段。

**3. 影响范围** 影响服务期及销转期学员的自动应答稳定性，部分学员未能及时收到课程相关回复。

**原因**

- **实际原因**：消息在短时间内集中进线，触发模型接口频率上限。
    
- **本质原因**：接口链路质检机制不健全，高并发时服务不稳定无法应对。
    

**解决方案**

- 临时切换至 GPT-5 模型以保障基本回复连续性；
    
- 同步恢复 Gemini 主模型作为核心应答引擎，在确保质量的前提下强化质检效率；
    
- 后续优化接口链路，加入请求限流与容错机制，从源头降低报错与频率超限风险。

    1. **AI总结需覆盖DAY1–DAY14全量**。
        
    2. **引入时间维度 + 课前/课中/课后**的条件化回复与点评逻辑。

#### AI应答与点评规则

1. **核心应答规则**
    

前提：需识别 **用户当前所处的课** + **时间窗（课前/课中/课后）**。

- **课前（当日上课前）**：引导用户当日按时上课。核心目标=**引导当晚到课**（或次日晨练，视DAY而定）。
    
- **课中**：核心目标=**引导继续上课/复连**，处理掉线与“没空”场景（给回放承诺）
    
    - 若直播掉线 → 引导继续进入课堂。
        
    - 若反馈“没时间/在上班” → 引导次日或当日课后看回放。
        
- **课后**：
    
    - （当日课程结束后）：
        
        - 若**未完课**：引导**看回放/补课**。
            
        - 若**已完课且有效到课**：收集**客户感受**→**AI点评**（**正向**=鼓励+复习/到课引导；**负向**=原因解释→危机/蓝图→建议与作业）提示 **第2天晨练**（如D1 → D2晨练）。。
            

2. **破冰分流**
    
    1. **已破冰**：直接**引导上课**。
        
    2. **未破冰**：**痛点解释**→价值锚定→上课引导（必要时下“轻危机”，尤其前序无痛点表达时）。
        
3. **正/负向感受分流**
    
    1. **正向**：强化收益→复习/到课提醒（不做“危机蓝图”）。
        
    2. **负向**：**原因解释**→**危机提醒**→**蓝图与课程价值**→**到课引导**。

    function formatTimestamp(tsStr) {
      // 转为数字并添加8小时（28800000毫秒）
      const timestamp = Number(tsStr) + 8 * 60 * 60 * 1000;
      const date = new Date(timestamp);
    
      const year = date.getUTCFullYear();  // 使用UTC方法确保一致性
      const month = padZero(date.getUTCMonth() + 1);
      const day = padZero(date.getUTCDate());
      const hour = padZero(date.getUTCHours());
      const minute = padZero(date.getUTCMinutes());
      const second = padZero(date.getUTCSeconds());
    
      return `${year}-${month}-${day} ${hour}:${minute}:${second}`;
    }
    
    function padZero(n) {
      return n < 10 ? '0' + n : n;
    }
    
    return {
      "Timestamp": formatTimestamp(timestamp)
    }
